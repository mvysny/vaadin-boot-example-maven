## Project overview

A minimal Vaadin Flow demo that runs on embedded Jetty via a plain `main()` method, using [Vaadin Boot](https://github.com/mvysny/vaadin-boot). No Spring/Spring Boot. Requires Java 21+. Vaadin version is pinned via the `vaadin.version` property in `pom.xml`; the Vaadin Boot library version is pinned separately on the `com.github.mvysny.vaadin-boot:vaadin-boot` dependency.

## Commands

Uses the Maven Wrapper — always invoke via `./mvnw`.

- **Run in development mode** (live-reload frontend via Vite): `./mvnw -C exec:java`
  - Visit <http://localhost:8080>. Press Enter in the console to stop. Dev mode uses `vaadin-dev` (optional dep) and starts Vite under the hood.
- **Run tests**: `./mvnw test`
- **Run a single test**: `./mvnw test -Dtest=MainViewTest#testGreeting`
- **Production build** (produces the runnable zip/tar.gz in `target/`): `./mvnw -C clean package -Pproduction`
  - `-Pproduction` is a Vaadin BOM profile that switches the frontend bundle to optimized/minified mode. Always use it for release artifacts.
- **Run production artifact**: `unzip target/*.zip -d dist && dist/bin/app`
- **Docker**: `docker build -t test/vaadin-boot-example-maven:latest .` then `docker run --rm -ti -p8080:8080 -m256m test/vaadin-boot-example-maven`

## Architecture

Four Java classes under `com.vaadin.starter.skeleton` carry the whole app:

- `Main` — entry point; `new VaadinBoot().run()` boots embedded Jetty with Vaadin.
- `AppShell` implements `AppShellConfigurator` and is the single place for app-wide concerns: `@PWA`, theme (`Lumo.STYLESHEET`), and global `styles.css`.
- `Bootstrap` is a `@WebListener ServletContextListener` — the hook for one-time init/teardown (DB pools, background executors). Tests invoke it manually (see below).
- `MainView` (`@Route("")`) is the UI.

Static web resources live under `src/main/resources/webapp/` (served as the webapp root) and `src/main/frontend/` (Vite-managed). Everything in `src/main/frontend/generated/` is regenerated by the Vaadin Maven plugin — do not hand-edit.

## Testing (Karibu-Testing, browserless)

Tests use [Karibu-Testing](https://github.com/mvysny/karibu-testing) to mock Vaadin in-JVM — no browser, no Jetty. Pattern (see `MainViewTest`):

1. `@BeforeAll`: `Routes().autoDiscoverViews("com.vaadin.starter.skeleton")` once per run (classpath scan is expensive), and call `new Bootstrap().contextInitialized(null)` so production init runs in tests too.
2. `@BeforeEach MockVaadin.setup(routes)` / `@AfterEach MockVaadin.tearDown()`.
3. Drive UI with `_get`, `_setValue`, `_click`, and assert on notifications with `expectNotifications(...)`.

When adding new views, they're picked up automatically via the `autoDiscoverViews` scan as long as they're under that package.

## Production packaging

`maven-assembly-plugin` with `src/main/assembly/zip.xml` produces `target/*.zip` and `target/*.tar.gz` containing `lib/` (all runtime jars, `vaadin-dev` excluded) and `bin/app` (the launcher script from `src/main/scripts/app`, made executable). The Dockerfile consumes the tarball from a multi-stage build.

## Vaadin version upgrades

Bumping `vaadin.version` also bumps the `vaadin-maven-plugin` (same property). When upgrading, also check whether `com.github.mvysny.vaadin-boot:vaadin-boot` and `karibu-testing-v24` have matching newer releases — they're versioned independently of Vaadin.
